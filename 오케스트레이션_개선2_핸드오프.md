# 오케스트레이션 개선 2 (코덱스 구현용)

3가지: (1) 백그라운드 실행, (2) 검토 게이트(승인 후 실행), (3) 중복 행 버그 수정.
함수/변수명은 현 코드에 맞게 조정.

---

## 0. 진행 상태를 전역으로 (공통 기반)

진행을 모달이 아니라 `state`에 둔다 → 어디서든 렌더 가능, 모달 닫아도 유지.

```js
state.orch = state.orch || { running:false, goal:"", items:[], summary:"", startedAt:0 };
// items: [{ key, employeeId, name, subtask, status, text, needsReview }]
//   status: "queued" | "review" | "running" | "done" | "error" | "skipped"
```
- `items[i].key = employeeId + "#" + i` (고유키 — 중복행 방지의 핵심)

## 1. 중복 행 버그 수정 (먼저)

원인: onUpdate가 "start"에서 한 줄, "done"에서 또 한 줄 push.
해결: **key로 기존 항목을 찾아 status/text만 갱신**, 항상 `state.orch.items`로부터 렌더.

```js
function upsertItem(key, patch){
  const it = state.orch.items.find(x=>x.key===key);
  if (it) Object.assign(it, patch); 
  renderOrchPanel();           // items 전체를 다시 그림(추가 X, 갱신 O)
}
function renderOrchPanel(){
  const done = state.orch.items.filter(i=>i.status==="done").length;
  const run  = state.orch.items.filter(i=>i.status==="running").length;
  const err  = state.orch.items.filter(i=>i.status==="error").length;
  // 헤더: 완료 done · 처리 중 run · 오류 err
  // 목록: items.map → 한 직원당 '한 줄', status 뱃지만 바뀜
}
```
→ 한 직원은 항상 한 줄, 뱃지가 처리중→완료로 바뀜.

## 2. 백그라운드 실행 (창 닫아도 계속)

핵심: 실행 루프는 DOM이 아니라 `state.orch`만 본다. 모달은 그 상태의 '뷰'일 뿐.

```js
async function runOrchestration(goal){
  state.orch = { running:true, goal, items:[], summary:"", startedAt:Date.now() };
  const plan = await planTasks(goal, employees);
  state.orch.items = plan.map((p,i)=>({
    key:p.employeeId+"#"+i, employeeId:p.employeeId,
    name:(getEmployee(p.employeeId)||{}).name||p.employeeId,
    subtask:p.subtask, status: p.needsReview ? "review" : "queued",
    text:"", needsReview:!!p.needsReview
  }));
  renderOrchPanel(); renderOrchBadge();

  for (const it of state.orch.items){
    if (it.status==="review") continue;          // 검토 대기는 건너뜀(2단계에서 승인 시 실행)
    await execItem(it);
  }
  state.orch.running = false;
  // 검토 대기가 남았으면 알림, 아니면 요약
  if (!state.orch.items.some(i=>i.status==="review")) await finishOrch();
  renderOrchBadge();
}

async function execItem(it){
  const emp = getEmployee(it.employeeId); if(!emp) return;
  upsertItem(it.key,{status:"running"}); setEmployeeStatus(emp.id,"working");
  try{ const text = await requestEmployeeReply(emp, it.subtask);
       upsertItem(it.key,{status:"done", text});
       registerBoardTask(emp, it.subtask);
  }catch(e){ upsertItem(it.key,{status:"error", text:String(e.message||e)}); }
  saveState();
}

async function finishOrch(){
  try{ state.orch.summary = await summarizeOrchestration(state.orch.goal, doneResults()); }
  catch(_){}
  renderOrchPanel();
  showToast("분배 완료 — 결과를 확인하세요");   // 다른 화면에 있어도 알림
}
```

UI 변경:
- 모달의 닫기(X)는 **루프를 멈추지 않음**(그냥 뷰만 숨김). 루프는 state로 계속 진행.
- 상단 툴바에 **진행 배지**: `renderOrchBadge()` → "🛠 분배 2/5" 클릭 시 모달 다시 열기.
- 완료 시 `showToast`로 알림 + 배지를 "결과 보기"로.
- (선택) `localStorage`에 `state.orch` 저장 → 새로고침해도 결과 유지.

## 3. 검토 게이트 (승인 후 실행)

### 3-1. 어떤 과제가 검토 대상인가
방법 A) 매니저가 판단 — planTasks의 JSON에 `needsReview` 추가:
```
[{"employeeId":"report-writer","subtask":"...","needsReview":true}, ...]
```
매니저 system 프롬프트에 한 줄 추가:
> "외부 공개·발표·보고·고객 전송처럼 사람이 확인해야 하는 과제는 needsReview:true, 내부 초안성 작업은 false로 표시하라."

방법 B) 규칙 — 특정 직원/키워드는 자동 검토:
```js
const REVIEW_IDS = ["report-writer","ax-pm","brand-strategist"]; // 보고/대외
const REVIEW_KWS = ["발표","보고","대외","고객","제출","공개"];
function needsReview(item, goal){
  return REVIEW_IDS.includes(item.employeeId) || REVIEW_KWS.some(k=>(item.subtask+goal).includes(k));
}
```
(A+B 병행 권장: 모델이 놓쳐도 규칙이 보완)

### 3-2. 동작
- `status:"review"`인 항목은 자동 실행 안 함 → 목록에 **[승인] [수정] [건너뛰기]** 버튼.
- **승인** → `execItem(it)` 실행.
- **수정** → subtask 텍스트 편집 후 승인.
- **건너뛰기** → `upsertItem(it.key,{status:"skipped"})`.
- 검토 대기 항목 처리가 끝나면 `finishOrch()`로 요약.

```js
function approveItem(key){ const it=state.orch.items.find(x=>x.key===key); if(it){ it.status="queued"; execItem(it).then(maybeFinish);} }
function editItem(key,newSub){ const it=state.orch.items.find(x=>x.key===key); if(it) it.subtask=newSub; }
function skipItem(key){ upsertItem(key,{status:"skipped"}); maybeFinish(); }
function maybeFinish(){ if(!state.orch.items.some(i=>["review","running","queued"].includes(i.status))) finishOrch(); }
```

UI: 검토 대기 항목은 노란 "검토 필요" 뱃지 + 버튼들. 내가 승인해야 그 직원이 '실시'함.

## 6. 출력 간결화 (결과가 너무 길어 못 읽는 문제) — 우선 적용

증상: 직원 답변/요약이 13개 항목짜리 장문이라 읽기 어려움.

### 6-1. 모델이 짧게 답하게 (가장 효과 큼)
worker.js `handleAgent`에서 system에 형식 규칙을 덧붙이고 max_tokens를 낮춘다:
```js
const BREVITY = "\n\n[출력 형식 규칙] 한국어로 핵심만. 불릿 3~5개, 각 항목 한 줄(최대 한 문장). " +
  "서론·결론·일반론 금지. 전체 8줄 이내. 표/장문 설명 금지.";
const r = await env.AI.run("@cf/meta/llama-3.1-8b-instruct-fp8", {
  messages: [
    ...(system ? [{ role:"system", content: system + BREVITY }] : [{role:"system",content:BREVITY}]),
    { role:"user", content: user },
  ],
  max_tokens: 380,          // 800 → 380 (장문 방지)
});
```
- 매니저(planTasks)는 JSON이라 BREVITY 영향 적음(그대로 OK). 직원/요약 호출에만 적용되면 충분.
- 요약(summarizeOrchestration)도 "전체 10줄 이내, 핵심 체크리스트만"으로 지시.

### 6-2. 직원 페르소나에도 한 줄 추가(선택)
office-data.js 각 `prompt.system` 끝에 공통으로: "항상 간결하게, 불릿 5개 이내로 답한다." (또는 buildEmployeePrompt에서 일괄 append)

### 6-3. UI에서 길면 접기
긴 답변은 카드에서 자르고 '더보기'로 펼치기:
```css
.orch-card .answer, .chat-msg.ai{
  max-height:160px; overflow:hidden; position:relative;
}
.orch-card .answer.expanded{ max-height:none; }
```
```js
// 답변 길면 '더보기/접기' 버튼 추가, 클릭 시 .expanded 토글
```
또는 간단히 `max-height:220px; overflow:auto`로 스크롤 박스 처리.

### 적용 우선순위
6-1(max_tokens↓ + 형식규칙)만 적용해도 체감이 확 달라짐 → 그다음 6-3(접기).

## 7. 업무 항목 클릭 → 상세 보기 (완료/처리중 내용 확인)

증상: 진행 목록·할 일판의 업무를 눌러도 내용을 못 봄.
목표: 항목 클릭 → 그 업무의 **지시·전체 답변·상태**를 상세로 표시.

- 데이터는 이미 있음: `state.orch.items[i] = { key, name, subtask, status, text }`.
- 각 행에 `data-orch-key="<key>"` 부여 + 클릭 핸들러:
```js
document.querySelector(".orch-results").addEventListener("click", e=>{
  const row = e.target.closest("[data-orch-key]"); if(!row) return;
  const it = state.orch.items.find(x=>x.key===row.dataset.orchKey); if(!it) return;
  openOrchDetail(it);   // 모달/패널에 it.name, it.subtask, it.text(전체), it.status 표시
});
```
- `openOrchDetail`: 채팅 말풍선 스타일(다크 배경)로 전체 답변 표시 + 닫기.
- 할 일판(task)에서도 보이게 하려면: 오케스트레이션으로 만든 task에 `resultText`를 같이 저장하고, 카드 클릭 시 동일 상세 표시.
- 처리 중(running)·검토 대기(review) 항목도 클릭 가능 — 현재 상태/지시를 보여주면 됨.

## 8. 우측 상단 "전체화면(키워보기)" 버튼

목표: 회사 건물 화면을 크게 보기.

- 상단 툴바 우측에 버튼 추가:
```html
<button id="fsBtn" class="nav-button" title="크게 보기" aria-label="전체화면">⛶</button>
```
- Fullscreen API로 건물 무대(또는 앱 루트)를 전체화면:
```js
const fsTarget = document.querySelector(".building-stage") || document.querySelector(".game-root");
document.getElementById("fsBtn").addEventListener("click", ()=>{
  if (!document.fullscreenElement) fsTarget.requestFullscreen?.();
  else document.exitFullscreen?.();
});
```
- 전체화면 중엔 아이콘을 ⤢↩ 등으로 토글. (모바일 Safari는 requestFullscreen 제한 — 그땐 CSS 확대 클래스로 대체: 루트에 position:fixed; inset:0; z-index:9999 토글)

## 9. 하단 'LIVE OFFICE' 박스 — 불투명 고동색 배경

증상: 배경이 밝아 글씨가 안 보임. → 불투명한 진한 고동색 + 밝은 글씨.

```css
body.v2-active .bottom-info-strip{
  background: linear-gradient(180deg, #3a2415 0%, #241509 100%) !important; /* 불투명 고동색 */
  border: 1px solid rgba(214,170,92,0.35) !important;
  color: #f3e6c8 !important;
}
body.v2-active .bottom-info-strip, 
body.v2-active .bottom-info-strip *{ color:#f3e6c8 !important; }
body.v2-active .bottom-info-strip strong{ background:#caa15a !important; color:#241509 !important; } /* LIVE OFFICE 라벨 */
body.v2-active .bottom-info-strip .office-census{ color:#ffe6a6 !important; }   /* 집계 줄 강조 */
```
- 색감: `#3a2415`(고동) → `#241509`(더 진한 고동) 그라데이션. 더 진하게 원하면 `#2a160b` 단색도 OK.
- 글씨 `#f3e6c8`(밝은 크림)로 대비 확보.
